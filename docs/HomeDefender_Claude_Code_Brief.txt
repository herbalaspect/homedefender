Should You Move Off Webflow?
Worth pausing on before committing. The right answer depends on what matters more after launch.
The case for staying on Webflow
Webflow handles hosting, CDN, SSL, image optimization, and sitemap generation automatically — all of those become your problem on a custom build.
Webflow's visual CMS lets non-technical people edit copy, swap images, and add city pages without writing code.
Your existing SEO bugs aren't Webflow's fault — they're content bugs that would exist on any platform. Fixing them in Webflow is faster than rebuilding.
If you'll rarely touch the site after launch, the platform doesn't matter much.
The case for moving to Next.js + Claude Code
Per-page meta description and schema markup is awkward in Webflow at scale. With 45+ pages each needing custom schema, code-driven gets tighter.
11 city landing pages can be generated programmatically from a data file. In Webflow that's CMS Collections setup; in Next.js it's 30 lines of code.
Page speed on Webflow is decent but rarely great. Next.js with proper optimization scores nearly perfect Core Web Vitals.
Once the project is in a git repo, you have version history, can roll back, and can A/B test changes branch-by-branch.
Iteration speed with Claude Code is genuinely faster than building in Webflow's UI for someone comfortable with code.
Decision criteria
Move to Claude Code if:
(1) You want the new build live faster than Webflow CMS work would take, (2) you're comfortable that future copy edits will require code changes, (3) you have time to migrate cleanly without losing search rankings.

Stay on Webflow if:
(1) You'll be updating content frequently, (2) you want a non-technical person to manage it, (3) you don't want to deal with hosting, deploys, and DNS yourself.

The rest of this document assumes the move is happening. If after reading the migration plan you decide it's more work than fixing Webflow, that's a legitimate conclusion — and you have all the content docs you need to do it in Webflow instead.
Recommended Tech Stack
Stack chosen for: SEO performance, Claude Code compatibility, low ongoing cost, fast deploys, and ability for a non-developer to make small edits later if needed.

Framework
Next.js 15 (App Router)
Best-in-class SEO out of the box. Static generation for fast page loads. Built-in image optimization. Per-page meta tags and structured data are first-class.
Styling
Tailwind CSS
Most-used styling system, well-documented, AI-friendly. Easy to match your existing brand colors (navy #1A2E3B, green #3A7D5C).
Content
MDX + data files
Blog posts in MDX (Markdown + components). City pages and service pages generated from TypeScript data files. No CMS dependency, no monthly fee.
Forms
Formspree or Resend
Contact form needs a backend. Formspree is simplest ($10/mo) — Resend ($20/mo) gives more flexibility for follow-up sequences. Pick one based on whether you need email automation.
Hosting
Vercel
Free tier sufficient for a marketing site at your traffic level. Auto-deploys on git push. Built-in CDN. Native Next.js host.
Analytics
Google Analytics 4 + Vercel Analytics
GA4 for marketing analytics and Search Console integration. Vercel Analytics for Core Web Vitals monitoring (free).
Domain
Keep current registrar
Just point DNS to Vercel after launch. Don't transfer the registrar — too much risk during migration.

Estimated monthly cost
Vercel: $0 (Hobby tier covers it; upgrade only if traffic grows past 100k visits/month)
Domain: ~$15/year (whatever you pay now)
Formspree (or Resend): $10–20/month
Total: ~$10–20/month vs. current Webflow cost (typically $23–39/month for CMS plans)

Why this structure
All strategy docs in /docs/ means Claude Code can reference them directly when generating page copy. The model reads markdown easily; for the docx files, also save markdown versions if you want maximum AI usefulness.
Page content lives in /content/ as TypeScript data files, not hardcoded into pages. Updating a city's testimonial means editing one line in content/cities.ts — not searching through JSX.
Blog posts in MDX means you can write blog content as Markdown with custom React components embedded. AI-friendly format.
Dynamic route /services/roofing/roof-replacement-[city] generates all 11 city pages from one template + the cities.ts data file.
/components/seo/ houses reusable schema components — drop a <LocalBusinessSchema /> on any page.

Claude Code Prompt Sequence
Use these prompts in order. Each builds on the previous one. Don't skip prompts — Claude Code's quality depends on having the foundation in place when later prompts run.
Pro tip: between prompts, commit and review what was generated. Catching mistakes early is much cheaper than catching them at deploy time.
Prompt 0 — Initialize the project
Purpose: Set up Next.js with the right configuration

Create a new Next.js 15 project for homedefenderusa.com. Use:
- App Router (not Pages Router)
- TypeScript
- Tailwind CSS
- src/ directory layout
- Import alias: @/*
 
After project creation, set up the folder structure exactly as specified in 
docs/claude-code-brief.md "Repository Structure" section. Create empty files 
where indicated. Read CLAUDE.md before doing anything else and confirm you 
understand the project conventions before writing any code.

SEO Migration Strategy
This is the section that determines whether the migration helps or hurts your search rankings. Done right, you preserve everything you've built and improve from there. Done wrong, you lose months of ranking progress.
The biggest risk in migration
If a page exists in Google's index pointing to homedefenderusa.com/X and you launch a new site without that URL, Google sees a 404. Repeated 404s on pages that previously ranked tank rankings across the whole site. The fix is comprehensive 301 redirects from every old URL to its new equivalent — even URLs you forgot existed.
Step-by-step migration plan
#
Step
Detail
1
Pre-migration audit (1 week before launch)
Crawl your current Webflow site with Screaming Frog (free tier covers <500 URLs). Export every URL. Compare to Google Search Console's Pages report — note which URLs are indexed and getting traffic. This becomes your redirect map.
2
Build new site on a subdomain first
Deploy to staging.homedefenderusa.com (or a Vercel preview URL). Test everything: forms, schema validation (Google Rich Results Test), mobile, page speed. Do NOT point your main domain yet.
3
Set up 301 redirects in next.config.ts
Map every old URL to its new equivalent. URLs that don't change still need to be in the redirect file as a safety net. URLs that do change (like /testimonial → /testimonials) need explicit redirects.
4
Submit new sitemap to Google Search Console (still on Webflow)
Don't wait until launch. Google can pre-index your new URLs while the old ones are still live. Use Search Console's URL Inspection tool to request indexing on the staging URLs.
5
Cutover day: switch DNS
Point homedefenderusa.com to Vercel. DNS propagation takes 1–24 hours. During this window, both old and new sites will be hit by different users — that's normal.
6
Verify redirects working
Use a tool like httpstatus.io to check that every old URL returns a 301 to the new URL. Spot-check 20+ URLs manually.
7
Resubmit sitemap and request re-indexing
In Search Console: submit the new sitemap.xml URL, then use URL Inspection to request indexing on top 10 most-trafficked pages.
8
Monitor Search Console daily for 30 days
Watch for: 404 errors (means a redirect was missed), crawl errors, index coverage drops, ranking volatility. Most issues show up in week 1–2.
9
Cancel Webflow
Wait 30 days minimum after successful cutover. Then archive the Webflow site (do NOT delete) and cancel the subscription.
